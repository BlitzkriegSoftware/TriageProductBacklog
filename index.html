Overview: This process is generalized

This is a generic process, it can work with GitHub Issues, ADO, Jira, Buginizer, etc.

The implementation details therefore vary, but the idea of a minumum flow is the key here.

This flow is intended for projects that have external people "the community" who provide bug reports and feature requests (see #personas)

Personas: Whose who

Persona Purpose
Community (CMTY) Provide feedback in the form of bug reports or feature requests
Triage & Support Team (TAST) Does the initial triage, and also contributes to testing and bug bashing
Product Team: Bug Queue (PTBQ) Product Team has two queues, this is one the bug queue
Product Team: Feature Request Queue (PTFR) Product Team has two queues, this is the other the feature request queue

Labels

Labels AKA Tags are used in some systems as a surrogate for state or status as in ADO or Jira.

In many cases these can be used in combination. In any case, the idea is to clearly indicate to someone examining an item, or quering for items what comes next for the item.

Status Labels

In this guide, we're going to assume GitHub issues, and here are a list of labels (some are the standard or default available ones) we'll be using, others we're adding

Label Description
(blank) | new To be triaged
bug Something isn't working
documentation Improvements or additions to documentation
duplicate This issue or pull request already exists
enhancement New feature or request
invalid This doesn't seem right
investigate Team is investigating
question Further information is requested
wontfix This will not be worked on

Team Labels

This are in addition to the above labels

Label Team Name
(blank) | CMTY Community

PTBQ Product Team Bug Queue

Review After

These labels are used so we can query items to be reviewed after some date

Label Format: 

yyyyWW
where 
yyyy
is the 4 digit year, and 
WW
is the week number 
01-52
.

For example 

202515
The 15th week of 2025 runs from Monday, April 7, 2025, to Sunday, April 13, 2025.

Start Here

For a specific item (in GitHub an Issue), evaluate the following and select one of these next actions, this process is the same for new items or items due for review

  • Item is invalid/unintelligable. The term "unintelligible" means unable to be understood or comprehended.
  • Item has expired
  • Item is a bug
  • Item is a feature request
  • Item is a documentation request/fix/need-more-examples

Invalid

  1. Add label 
    invalid
    label
  2. Add a review-after label (I prefer 4 weeks from today) e.g., if it is 202532 now, then add 
    202536
  3. Add a detailed review (politely) indicating what else is needed to address the issue

Expired

The review-after period has expired, it was assigned to community (CMTY), and no response has been received to the 

invalid
label

  1. Remove all previous labels
  2. Add 
    wontfix
    label
  3. Add a brief review that the issue is closed until more information is received
