Overview: This process is generalized

This flow is intended for projects that have external people "the community" who provide bug reports and feature requests (see #personas) and is targeted at GitHub issues, so if using something else, so "tuning" may be required.

The implementation details therefore vary, but the idea of a minumum flow is the key here and is intended to give guidance to teams maintaining open source projects with active communities, so the frequency of the suggested activities may need to be adjusted based on the number of issues per week.

Personas: Whose who

Persona Label Purpose
Community P-CMTY Provide feedback in the form of bug reports or feature requests
Triage & Support Team P-TAST Does the initial triage, and also contributes to testing and bug bashing in support of the main developers AKA the "Product Team" who maintain two queues of work, see below. The TAST in addition, to triage, does light housekeeping including small bugs, documentation fixes, more tests, etc.
Product Team: Bug Queue P-PTBQ Product Team has two queues, this is one the bug queue
Product Team: Feature Request Queue P-PTFR Product Team has two queues, this is the other the feature request queue

Labels

Labels AKA Tags are used in some systems as a surrogate for state or status or other fields in ADO, Jira, etc.

In many cases these can be used in combination. In any case, the idea is to clearly indicate to someone examining an item, or quering for items what comes next for the item.

Team Labels

See Persona labels

Status Labels

In this guide, we're going to assume GitHub issues, and here are a list of labels (some are the standard or default available ones) we'll be using, others we're adding

Label Description
(no label) | new To be triaged
bug Something isn't working
documentation Improvements or additions to documentation
duplicate This issue or pull request already exists
feature New feature or request
fixed The item is resolved fixed!
invalid This doesn't seem right
investigate Team is investigating
tests Issues with tests, missing tests, requests for more tests, or other test cases
wontfix This will not be worked on

Priority / Severity Labels

These give some indication of the perceived urgency or value of the proposed change

Label Rating Typical Time to Fix
1-Critical This is urgent or the most severe, and must reflect that the system is profoundly broken unless fixed 2-3 Days unless fix is a breaking change
2-High Some major functionality is broken, on at least one of the supported platforms In the next release, unlesss security related, then ASAP.
3-Medium This is the most severe any issue can be if there is a functional work-around, and indicates that something is broken and should be fixed. This is the highes level typically assigned to requests for documentation fixes or more tests. Unless the documentation is just wrong (or missing key elements). In the next release ideally, the one after that if not possible.
4-Low This is an annoyance which must be routinely worked around, or requires more code than needed. When possible
5-Useful This is a Nice to have sometime in the future Whenever

Review After

These labels are used so we can query items to be reviewed after some date

Label Format: yyyyWW where yyyy is the 4 digit year, and WW is the week number 01-52.

For example 202515 The 15th week of 2025 runs from Monday, April 7, 2025, to Sunday, April 13, 2025.

Routines

This is the ideal flow

  1. Start of every week, run a query for any review-after label that is from this or previous weeks, and do the triage for all of those first
  2. Do initial triage new items every morning
  3. Finish up fixed items
  4. Do the investigation of each bug item
  5. Work on backlog assigned to TAST e.g., bugs, documentation, etc. tasks
  6. Weekly house keeping:
    • Do a daily report to stakeholders
    • Clean up expired branches
    • Query for "lost" items and triage or schedule them for triage
^top

Triage Items

For a specific item (in GitHub an Issue), evaluate the following and select one of these next actions, this process is the same for new items or items due for review

  • Item is invalid/unintelligable. The term "unintelligible" means unable to be understood or comprehended.
  • Item has expired e.g., has popped-up as the item came up on the query of review after items
  • Item is a bug
  • Item is a feature request
  • Item is a documentation request/fix/need-more-examples
  • Item is test related
  • Item is fixed
^top

Investigation Process

This is the process to investigate bugs

  1. Check to see if the bug is against a valid environment, e.g., OS, runtime, tool-chain, etc. if not, stop, and reject bug as wontfix using the Won't Fix Process.
  2. If the bug report does not indicate a specific release, test against latest release.
  3. Likewise, add the exact Operating System and Version into the bug.
  4. Pull a clean version of the code at the appropriate version
  5. Create a branch named per the shop standard in the GIT Repo. If there is not a branch standard, use this format, with all characters forced to lower case, and no Punctuation except underscore:
    {team-label}/{username}/{issue#}/{remark}.
    Examples:
    • tast/jdev/323/nullex
    • tast/ptst/123/repro
    • ptfr/jader/631/add_magic_stuff
  6. Write at least one test that attempts to duplicate the bug!
  7. Test
  8. Update item comments
  9. Decide what to do next:
    • Can't repoduce bug
    • Can repoduce bug

Flows (aka Procedures)

In alphabetical order

Bug

Bugs reports are processed like this:

  1. Remove all previous labels
  2. Add label bug label
  3. Add a review-after label of this week for triage
  4. Add any initial comments
  5. (next item)

Can repoduce bug

We can reproduce the reported bug.

  1. Remove all previous labels
  2. Add label bug
  3. Add comments about how bug could be solved, and why it was assigned to what team
  4. If bug is trivial, assign to TAST, if it is more complex, assign it to the PTBQ
  5. Give it an initial review-after label of 2-8 weeks depending on release schedule, and capacity (if in doubt use the default of 4 weeks)
  6. (next item)

Can't repoduce bug

We can't reproduce the reported bug. Oh, no.

  1. Remove all previous labels
  2. Add label CMTY
  3. Add label bug
  4. Add a review-after label (I prefer 4 weeks from today) e.g., if it is 202532 now, then add 4 weeks yielding 202536
  5. Add a detailed review (politely) indicating why we were unable to reproduce the bug, and ask for more details.
  6. Return to submitter
  7. (next item)

Documentation Request

  1. Remove all previous labels
  2. Add label documentation label
  3. Give it a team label of TAST in most cases, for complex scenarios, may need to escallate by assigning it to the PTFR
  4. Give it an initial review-after label of 4-8 weeks depending on release schedule, and capacity (if in doubt use the default of 4 weeks)
  5. Add comments and observations
  6. (next item)

Expired

The review-after period has expired, it was assigned to community (CMTY), and no response has been received to the invalid label

  1. Remove all previous labels
  2. Add wontfix label
  3. Add a brief review that the issue is closed until more information is received
  4. (next item)

Item is CLOSED

If this needs to be undone, follow the Kick-Back Process

Feature Request

  1. Remove all previous labels
  2. Add label feature label
  3. Give it a team label of PTFR
  4. Give it an initial review-after label of 4-8 weeks depending on release schedule, and capacity (if in doubt use the default of 4 weeks)
  5. Add comments and observations
  6. (next item)

The Product Team (PT) will work features along their own schedule. They may kick back items

Fixed

Item is resolved with a good outcome.

  1. Remove all previous labels
  2. Add label fixed
  3. Add a comment that item is fixed with a link to the release if it is code, and to the page(s) if documentation
  4. Delete related branches after the retention time (typically 90 days)
  5. (next item)

If this needs to be undone, follow the Kick-Back Process

Invalid

The item is bad, incomprehensible, or ...

  1. Add label invalid label
  2. Add a review-after label (I prefer 4 weeks from today) e.g., if it is 202532 now, then add 4 weeks yielding 202536
  3. Add a detailed review (politely) indicating what else is needed to address the issue
  4. (next item)

Kick Back by Product Team

Occasionally, the Product Team (PT) will need to kick-back an item from either the Bug or Feature Request Queue. The process is as follows:

  1. Remove all previous labels
  2. Add label invalid label
  3. The product team owes the TAST team good comments about what is missing, so they know what to do next.
  4. Add a review-after label of this week for re-review
  5. (next item)

Tests

This is the process for more tests requested. Tests are an idea way to demo code.

  1. Remove all previous labels
  2. Add label tests label
  3. Give it a team label of TAST in most cases, for complex scenarios, may need to escallate by assigning it to the PTFR
  4. Give it an initial review-after label of 4-8 weeks depending on release schedule, and capacity (if in doubt use the default of 4 weeks)
  5. Add comments and observations
  6. (next item)

Won't Fix aka Rejected

The bug is invalid, the feature is rejected, etc.

  1. Remove all previous labels
  2. Add label wontfix label
  3. Add a comment as to why.
  4. Return to submitter
  5. (next item)

If this needs to be undone, follow the Kick-Back Process
